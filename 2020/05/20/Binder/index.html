Binder

Binder驱动

binder驱动会将自己注册为一个misc device，并向上层提供一个/dev/binder节点，binder节点并不对应真正的硬件设备，Binder驱动运行在内核态，提供了open(),ioctl(),mmap()等常用的对文件的操作

binder_init()，注册misc device， 创建/dev/binder设备节点

binder_open(), 获取Binder Driver的你文件描述符

binder_mmap(), 在内核分配一块内存，用于存储数据 （128*1024byte）

binder_ioctl()，将IPC参数作为参数传递给Binder Driver，函数负责在两个进程间收发IPC数据和IPC reply数据

open

上层进程在访问Binder驱动时首先要做的就是打开/dev/binder设备节点。Binder驱动会在/proc系统目录生成各种管理信息，最终生成binder_proc作为管理数据的记录体（每个进程都有独立的记录），并对其进行初始化。之后用户的所有对Binder驱动的操作都将以binder_proc这个对象为基础

mmap

可以将设备指定的内存块直接映射到应用程序的内存空间中

假设有两个进程A，B，他们要通过binder进行进程间通信

1、应用程序通过mmap函数返回得到一个内存地址（虚拟内存），这个地址通过虚拟内存转换（分页，分段）后最终将指向物力内存中的某个位置

2、binder驱动，它也会有一个指针（binder_proc->buffer）指向某个虚拟内存地址，通过虚拟内存转换后，它和应用程序中指向的物力内存处于同一个位置

这样Binder和应用程序之间就拥有了公用的内存块，这样当他们造作各自的内存地址时就相当于是在同意个内存块中执行的

这样当Binder通过copy_from_user把进程A中的某段数据复制到其binder_proc->buffer所指向的内存空间中，应为binder_proc->buffer和B进程的内存区块是共享，这样B进程就可以直接访问到这段数据。即一次拷贝就完成了A，B进程间的数据共享

Binder驱动默认映射区块大小为128KB，最多只支持4MB空间的mmap操作，超过4MB时也只分配4MB

ioctl

这是Binder中接口函数中工作量最大的一个，它承载了主要的工作。

ioctl命令 数据类型 操作
BINDER_WRITE_READ struct binder_write_read 收发Binder IPC数据
BINDER_SET_MAX_THREADS __u32 设置Binder线程最大个数
BINDER_SET_CONTEXT_MGR __s32 设置Service Manager节点
BINDER_THREAD_EXIT __s32 释放Binder线程
BINDER_VERSION struct binder_version 获取Binder版本信息
BINDER_SET_IDLE_TIMEOUT __s64 没有使用
BINDER_SET_IDLE_PRIORITY __s32 没有使用

最重点的是读写操作BINDER_WRITE_READ。

需要注意的是他还提供了一个命令BINDER_SET_CONTEXT_MGR，是将该Binder Server设置为Service Manager节点，该命令为ServiceManager独有的。

ServiceManager（Binder Server）

位置：/frameworks/native/cmds/servicemanager/service_manager.c

ServiceManager在显示网络中相当于我们的DNS服务器，服务端向其注册自己正式的地址，客户端向其查询并获取对应服务。

就像DNS服务器一样，ServiceManager必须在用户使用Binder之前就准备就绪，所以它是由init程序在解析init.rc时启动的，一旦ServiceManager发生问题后重启，其他的系统程序zygote，media，surfaceflinger和drm也会被重新加载。

一、ServiceManager启动流程

1、打开Binder设备，做好初始化

2、将自己设置为Binder大管家（BINDER_SET_CONTEXT_MGR）

3、通知Binder驱动，告知其大管家已就位，等待命令，进入主循环，获取并执行指令。

二、主循环中做了什么

它会不断的从Binder驱动中通过ioctl函数的BINDER_WRITE_READ取出消息（命令）并进行处理，直到收到退出的消息。

在BR_TRANSACTION命令中主要处理以下功能

1、注册Binder Server以供查询

2、应用程序请求查询并获取Binder Server所对应的句柄

binder_prase方法中的命令

命令 功能
SVC_MGR_GET_SERVICE 根据Server名称查找它的handle值
SVC_MGR_CHECK_SERVICE 根据Server名称查找它的handle值
SVC_MGR_ADD_SERVICE 用于注册一个Binder Server
SVC_MGR_LIST_SERVICES 获取列表中所有的Server

ServiceManager其内部维护了一个svclist列表，用于储存所有Server相关信息（以svcinfo为数据结构），查询和注册都是基于这个表展开的。

在想Binder驱动发送BINDER_WRITE_READ命令读取消息时，如果没有消息就会入休眠等待。

开启循环时将svcmgr_handler作为binder_handler func参数传入

Binder架构设计思路

访问ServiceManager需要哪些步骤

1、打开Binder设备

2、执行mmap

3、通过Binder驱动向SM发送请求

4、获取结果

具体问题：

1、向SM发送请求的Binder Client可能是App应用，所以必须提供java层接口（java接口实现）

2、每个Binder Client在使用SM时都要亲力亲为所有的步骤，未免过于繁琐。（封装易于使用）

3、如果应用每次使用SM服务（或其他Binder Server）都用完成打开Binder设备，执行mmap，过于消耗系统资源（每个进程只允许打开一次Binder设备，切只进行一次内存映射，所有需要使用Binder驱动的线程共享这一资源）

ProcessState和IPCThreadState简介

ProcessState：

​ 专门管理每个应用进程中的Binder操作，切暴露出执行Binder驱动的一系列命令给上层用户

IPCThreadState：

​ 保证进程中的每个线程都有于Binder驱动进行通信的能力

​ binder_proc中有threads红黑树用于管理该进程中所有线程的IPC业务，基于Binder的IPC通信是阻塞的，这样保证个别thread在进行进程间通信是不会卡死整个操作系统

Proxy

通过ProcessState和IPCThreadState就能通过BINDER_WRITE_READ等Binder支持的命令对其进行交互，从而获得到SM提供的服务，但是这样过于繁琐了，所以就对其进行了封装。

IPCThreadState

1、保证IPCThreadState在线程中唯一，且其他线程不可使用

2、真正的完成与底层BinderDriver间的通信工作

为了保障其他线程不能使用，就需要TLS（Thread Local Storage）机制，它能保障某个变量仅在自己的线程内访问有效，在其他线程中得到的是这个变量的独立复本，互不干扰。

匿名服务

不在ServiceManager中进行注册的服务，只能通过一个实名服务作为中介获取到匿名服务的代理对象，然后操作匿名服务

典型的例子在WindowManagerService中，我们在希望与WindowManagerService进行交互时，是不能直接调用WindowManagerService中的方法的，因为WindowManagerService是系统中的重要服务，他需要与系统中的其他模块进行合作，所以不会把所有方法都暴露给我们，我们只有通过获取到他内部的Session对象，从而间接的通过与Session对象交互来达到目的，Session对象继承自IWindowSession.Stub，且其在new的时候会在构造方法中持有WindowManagerService的引用，当Session的引用作为Parcel被传输给客户端是，因为Binder驱动的路过Binder对象即绑定机制，会在Binder驱动服务端进程中生成对应的node节点，这样在传输给Client时，我们获得到的其实就是Session对象的代理对象，这样就可以直接跨进程访问Session，从而间接的访问WindowManagerService
